// 01 · The Problem Space

What Vector RAG Actually Solves
— And Where It Breaks

Vector RAG was invented to solve the context window problem. But the solution introduced a new, subtler failure mode.

The Real Problem
Context Window Overflow

A 500-page financial report contains ~250,000 tokens. Modern LLMs can't consume that in one shot. The answer: chunk it, embed it, retrieve top-k.

500-page PDF
→
chunk
→
embed
→
top-k retrieve
The Hidden Cost
Similarity ≠ Relevance

When an analyst asks about EBITDA, every section mentioning "earnings" scores high cosine similarity — but only one defines the specific calculation in scope. Vector search can't reason about that.

"EBITDA calc?"
→
12 similar chunks
→
wrong answer
⚠
Core Insight

Chunking destroys the document's hierarchy. A footnote referencing Appendix B, a table nested under Section 4.1, a definition that overrides an earlier one — all of this becomes invisible to cosine distance. The document's structure carries meaning that embeddings cannot encode.

// 02 · The PageIndex Approach

How PageIndex Works

Instead of embedding chunks into a vector space, PageIndex builds a living tree of the document's structure — then lets an LLM reason its way to the answer.

01
🌳
Tree Index Generation

The document is parsed into a hierarchical JSON tree — titles, sections, subsections, page ranges, and LLM-generated summaries. Structure is preserved, not flattened.

02
🧠
In-Context Reasoning

The tree index lives in the LLM's active context window. During retrieval, the model reasons over node titles and summaries to navigate to the right section — like a human using a table of contents.

03
🎯
Precise Extraction

Once the target node is identified, only that section's text is fetched. Bounded by max_page_num_each_node and max_token_num_each_node — context stays manageable.

💡
Context Cost Is Front-Loaded, Not Eliminated

PageIndex doesn't escape the context window problem — it relocates it. Index building makes many LLM calls: TOC detection per page, section verification (parallel async), summary generation per node, fix/retry calls via verify_toc + fix_incorrect_toc_with_retries (page_index.py:870). The retrieval time cost is low — but the build-time cost is significant.

// 04 · The Honest Tradeoffs

What You're Actually Trading

PageIndex is not a free lunch. Understanding the real costs is critical before adoption.

⬡ GPT Dependency — Critically Deep

01
TOC Extraction
GPT identifies section boundaries and headings from raw page text. A hallucination here sets the wrong page range for an entire branch.
02
Section Verification
Parallel async GPT calls validate each detected section. verify_toc + fix_incorrect_toc_with_retries exist because this fails in practice.
03
Summary Generation
Each node gets an LLM-generated summary. The retrieval reasoning depends on the quality of these summaries — garbage in, garbage out.
04
Retrieval Reasoning
At query time, GPT navigates the tree. Misreading a summary can send the model down the wrong branch entirely.
⚠
IF GPT HALLUCINATES OR MISIDENTIFIES A SECTION START PAGE,
THE ENTIRE TREE CAN BE STRUCTURALLY WRONG.
THE QUALITY CEILING OF THIS SYSTEM IS BOUNDED BY HOW WELL GPT UNDERSTANDS DOCUMENT STRUCTURE.
⚖
The Core Trade

PageIndex trades cheap indexing + unreliable retrieval for expensive indexing + precise retrieval. The cost is front-loaded at build time through many LLM calls. The benefit is measurable: reasoning-based navigation finds the right section, not just the similar one. For high-stakes documents where accuracy matters more than indexing speed — financial filings, legal contracts, technical manuals — that trade is often worth it.

// 05 · Bottom Line

PageIndex does one thing better than vector RAG:
it retrieves the right section, not the similar section.
The cost is higher LLM dependency during index build,
and a failure mode that corrupts the entire tree — not just one result.

Use it when retrieval accuracy matters more than indexing cost.
Don't use it as a drop-in replacement for every RAG use case.
