01 · The Problem

What Vector RAG Solves —
And Where It Breaks

Vector RAG was built to solve the context window problem. But the fix introduced a subtler, harder-to-debug failure: semantic similarity is not the same as relevance.

The Real Problem

Context Window Overflow

A 500-page financial report contains ~250,000 tokens. LLMs can't consume that in one shot. The standard fix: chunk it, embed it, retrieve the top-k most similar chunks at query time.

500-page PDF→ Chunk→ Embed→ Top-k retrieve
The Hidden Cost

Similarity ≠ Relevance

Ask "What is the EBITDA calculation?" — every section mentioning earnings scores high cosine similarity. But only one defines the specific calculation relevant to the question. Vector search cannot reason about that distinction.

"EBITDA calc?"→ 12 similar chunks→ Wrong answer
⬡ Core Insight

Chunking destroys a document's hierarchy. A footnote referencing Appendix B, a table nested under Section 4.1, a definition that overrides an earlier one — all this context becomes invisible to cosine distance. Document structure carries meaning that embeddings cannot encode.

02 · The Approach

How PageIndex Works

Instead of embedding chunks into a vector space, PageIndex builds a hierarchical tree of the document's natural structure — then lets an LLM reason its way to the correct answer.

🌳

01 · Tree Index Generation

The document is parsed into a hierarchical JSON tree — sections, subsections, page ranges, and LLM-generated summaries per node. Document structure is preserved, not flattened into arbitrary chunks.

🧠

02 · In-Context Reasoning

The tree index lives in the LLM's active context window. At retrieval time, the model reads node titles and summaries to navigate toward the right section — exactly like a human expert using a table of contents.

🎯

03 · Precise Extraction

Once the target node is identified, only that bounded section is fetched — controlled by max_page_num_each_node and max_token_num_each_node. Context stays manageable and fully traceable.

⬡ Important — Cost Is Front-Loaded, Not Eliminated

PageIndex doesn't escape the context window problem — it relocates it to build time. Index building makes many LLM calls: TOC detection per page, parallel async section verification, summary generation per node, and fix/retry calls via verify_toc + fix_incorrect_toc_with_retries (page_index.py:870). Retrieval is fast — but the build-time cost is significant.

04 · Honest Tradeoffs

What You're Actually Trading

PageIndex is not a free lunch. Understanding the real costs is essential before adopting it in production systems.

GPT Dependency — Critically Deep

01
TOC Extraction

GPT identifies section boundaries and headings from raw page text. A hallucination here sets the wrong page range for an entire branch of the tree.

02
Section Verification

Parallel async GPT calls validate each detected section. verify_toc + fix_incorrect_toc_with_retries logic exists precisely because this fails in practice.

03
Summary Generation

Each node gets an LLM-generated summary. Retrieval reasoning depends entirely on summary quality — garbage in, garbage out.

04
Retrieval Reasoning

At query time, GPT navigates the tree by reading node summaries. Misreading a summary sends the model down the entirely wrong branch.

⚠
If GPT hallucinates or misidentifies a section start page, the entire tree can be structurally wrong. The quality ceiling of this system is bounded entirely by how well GPT understands your document's structure.
⚖ The Core Trade

PageIndex trades cheap indexing + unreliable retrieval for expensive indexing + precise retrieval. Build-time cost is real and significant. The benefit is measurable: reasoning-based navigation finds the right section, not just the similar one. For high-stakes documents where accuracy outweighs indexing speed — financial filings, legal contracts, technical manuals — that trade is often worth making.

05 · Bottom Line

PageIndex does one thing better than vector RAG:
it retrieves the right section, not the similar section.
The cost is deeper LLM dependency at index build time,
with a failure mode that corrupts the entire tree — not just one result.

Use it when retrieval accuracy matters more than indexing cost. Don't treat it as a drop-in replacement for every RAG use case.
