01 · System Overview

What PageIndexJ Does

A single Java-based engine that reads a document, calls the OpenAI API across up to 9 structured steps, and writes a hierarchical JSON tree to disk. No database. No vector store. No network dependency beyond OpenAI.

Input

PDF or Markdown

Provide a file path to a .pdf or .md file plus optional config overrides. That's it — no pre-processing, no chunking, no embedding model needed.

Processing

9-Step LLM Pipeline

The engine parses pages, detects structure, verifies section locations with concurrent GPT-4o calls, builds a nested tree, and optionally generates per-node summaries.

Output

JSON Tree + Audit Log

A structured results/<name>_structure.json file containing the full hierarchy, and a timestamped logs/<name>_<ts>.json recording every decision.

⬡ System Boundary

External dependencies are exactly two: the OpenAI Chat Completions API (model: gpt-4o-2024-11-20, temperature: 0) and the local filesystem. No vector database, no embedding pipeline, no external services beyond OpenAI. All intelligence comes from GPT-4o; all state lives in the JSON output file.

02 · The Pipeline

9 Steps, Start to Finish

Every run follows the same deterministic sequence. Steps that involve OpenAI are clearly marked. Steps that run concurrent API calls are marked parallel.

00
Startup & Validation No API calls

Read config (merge config.yaml defaults with user overrides), validate the input file exists with the correct extension, load the API key from .env, and open the JSON audit log file for this run.

Fails if: file not foundFails if: API key missingFails if: wrong extensionUI state: VALIDATING
01
Parse Document into Pages No API calls

PDF: Apache PDFBox extracts text from every page and counts tokens using tiktoken. Output: List<PageEntry(text, tokenCount)>. Markdown: split on heading lines (#, ##, ###…) — each heading becomes a node with its text as content.

UI state: PARSINGProgress: pages X / NTypical output: 87 pages, 142,350 tokens
02
TOC Detection PDF only Up to 20 API calls

Scans pages 0 through toc_check_page_num (default: 20). For each page, asks GPT-4o: "Does this page contain a Table of Contents?" Stops scanning once a run of "yes" pages ends. If no TOC is found, skips directly to Step 4 Mode C.

UI state: TOC_SCANNINGProgress: page X / 20Can cancel
03
TOC Content Analysis 1 API call

Concatenates all TOC page text (cleans long dot sequences), then asks GPT-4o: "Does this TOC include page numbers?" The answer determines which extraction mode runs next: Mode A (TOC + page numbers), Mode B (TOC, no page numbers), or Mode C (no TOC).

UI state: TOC_ANALYZING
04
Structure Extraction — Modes A / B / C 5–15 API calls Can cancel

The core extraction step. Three paths depending on what Step 3 found. Mode A (TOC with page numbers): transforms TOC to JSON, computes physical page offset, fixes null entries. Mode B (TOC, no page numbers): walks the entire document in token-bounded chunks to locate each section. Mode C (no TOC): generates structure from scratch by reading chunks sequentially. See Extraction Modes section below for full detail.

UI state: EXTRACTINGProgress: chunks X / N
05
Verification 1 call per section (all concurrent)

For every extracted section, asks GPT-4o: "Does section '[title]' appear on this page?" All checks fire simultaneously. Computes an accuracy score. If accuracy > 60%: auto-fix incorrect items (Step 5a). If ≤ 60%: fall back to a simpler extraction mode. If 100%: proceed directly to Step 6.

UI state: VERIFYINGProgress: X / N · accuracy Y%50 concurrent calls for a 50-section doc
5a
Auto-Correction Up to 3× retry Concurrent calls

For each incorrect section: find the nearest correct sections above and below, retrieve all pages in that range, ask GPT-4o to pinpoint the exact start page. Runs concurrently for all incorrect items. Re-verifies after each round. Repeats up to 3 times.

UI state: FIXINGProgress: X corrected / N · attempt Y / 3
06
Tree Building No API calls

Pure logic. Adds a "Preface" node at page 1 if needed. Assigns start_index and end_index to every node. Converts the flat list into a nested tree using structure codes: "1" → root, "1.1" → child, "1.1.2" → grandchild. Concurrent checks determine whether section titles appear at the top of their page or partway through.

No API callsUI state: TREE_BUILDING
07
Recursive Sub-Indexing 5–15 calls per large node Concurrent

Any node spanning more than max_page_num_each_node pages (default: 10) AND more than max_token_num_each_node tokens (default: 20,000) gets re-indexed recursively. Steps 4–6 re-run on just that node's pages. Multiple large nodes are processed concurrently. Sub-nodes can themselves trigger further recursion.

UI state: SUB_INDEXINGe.g. "Chapter 3 — Methodology (pages 45–92)"
08
Enrichment — IDs, Summaries, Description 1 call per node (concurrent)

Four optional sub-steps controlled by config flags. 8a: assign breadth-first sequential node IDs ("0001", "0002"…). 8b: append raw page text to each node. 8c: generate a 2–3 sentence LLM summary per node (all concurrent). 8d: generate a single-sentence document description using the full tree.

UI state: SUMMARIZINGProgress: X / N summaries50 concurrent calls for a 50-node tree
09
Output No API calls

Serialize the tree with Jackson ObjectMapper, write to results/<filename>_structure.json. The log file is already written incrementally. Shut down the thread pool.

No API callsUI state: DONEThread pool: CPU × 4
03 · Extraction Modes

Three Paths Through Step 4

The path taken depends entirely on what's found during TOC detection. Mode A is fastest and most accurate. Mode C is the fallback when no structure exists upfront.

MODE A
TOC exists with page numbers · ~5–15 API calls
4A-1Transform TOC text into structured JSON with title, structure code, and page number per entry. If output is cut off by token limit, ask GPT-4o to continue until complete.
4A-2Compute physical page offset. The PDF's printed page "1" ≠ physical page 1 in the file. Scan post-TOC pages, match printed numbers to physical positions, find the most common offset, apply to every entry.
4A-3Fix null-page entries. For TOC items with no page number (footnotes, appendices), scan surrounding pages and ask GPT-4o to locate them.
MODE B
TOC exists, no page numbers · ~5–10 API calls
4B-1Transform TOC text into structured JSON with title and structure code only — no page field since the TOC has none.
4B-2Walk the entire document in token-bounded chunks (default: 20,000 tokens). For each chunk, ask GPT-4o which TOC sections start in this part and on which exact page. Accumulate answers across all chunks.
MODE C
No TOC found · ~5–10 API calls
4C-1Generate structure for the first chunk. Ask GPT-4o to produce a JSON tree showing sections, hierarchy codes (1, 1.1, 1.2, 2…), and physical page start for each section.
4C-2Continue structure for each subsequent chunk, passing the prior structure as context. Append new sections to the running list. For a 100-page doc: roughly 5–10 chunk calls total.
⬡ Why Three Modes Matter

Each mode has a different accuracy profile. Mode A is most reliable because GPT-4o is given concrete page numbers to verify against. Mode C is most error-prone because GPT-4o must infer structure purely from content — which is why Step 5 (Verification + Auto-Fix) exists as a safety net after all three modes.

05 · Configuration

All 8 Parameters

All parameters are optional with sensible defaults. Override any of them in config.yaml or pass as CLI arguments.

ParameterDefaultWhat It Controls
modelgpt-4o-2024-11-20Which OpenAI model handles every LLM call in the pipeline
toc_check_page_num20How many pages to scan when searching for an existing Table of Contents
max_page_num_each_node10A node spanning more pages than this threshold triggers recursive sub-indexing in Step 7
max_token_num_each_node20000A node with more tokens than this threshold triggers recursive sub-indexing (both conditions must be true)

Required Input

Exactly One File Path

Provide either pdf_path pointing to a .pdf file, or md_path pointing to a .md / .markdown file. Not both. Not neither. File must exist on the local filesystem.

Cost Tip

Tune for Your Use Case

Set if_add_node_summary=no to skip ~50 API calls for large documents. Increase max_page_num_each_node to prevent deep recursion on documents with very long chapters.

06 · Honest Tradeoffs

What You're Actually Trading

PageIndex is not a free lunch. Understanding the real costs is essential before adopting it in production.

GPT Dependency — Critically Deep Across 4 Stages

01
TOC Extraction (Steps 2–4)

GPT-4o identifies section boundaries, headings, and page numbers from raw page text. A hallucination here sets the wrong page range for an entire branch of the tree — and every downstream node inherits the error.

02
Section Verification (Step 5)

Concurrent GPT-4o calls validate each detected section start page. verify_toc + fix_incorrect_toc_with_retries (page_index.py:870) exist precisely because this fails in practice and needs up to 3 retry rounds.

03
Summary Generation (Step 8c)

Each node gets a 2–3 sentence LLM-generated summary. The quality of these summaries is critical — retrieval reasoning navigates the tree using summaries to decide which branch to follow. Garbage summaries → wrong branches.

04
Retrieval Reasoning (Downstream)

At query time, GPT-4o navigates the tree by reading node summaries. Misreading a summary, or encountering a node whose page range was mis-extracted, sends the model down the entirely wrong branch with high confidence.

⚠
If GPT-4o hallucinates a section start page during extraction, the entire tree can be structurally wrong. The quality ceiling of this system is bounded entirely by how well GPT-4o understands your specific document's structure and layout.
⚖ The Core Trade

PageIndex trades cheap, fast, imprecise retrieval (vector RAG) for expensive indexing + precise, reasoning-based retrieval. The build cost is real: ~133 API calls for a 100-page document. The benefit is measurable: reasoning-based navigation finds the right section, not just the similar one. For high-stakes, long-form documents — financial filings, legal contracts, technical manuals — where accuracy is more valuable than build speed, that trade is often worth making.

⬡ Concurrency Model

The thread pool size defaults to CPU cores × 4. The following operations fire all-at-once rather than sequentially: verification checks, auto-fix corrections, summary generation, sub-index processing, and appear_start checks. Each concurrent operation is one HTTP call to OpenAI. For the UI: these phases complete faster than their call count suggests, but exact duration depends on OpenAI response time — not document size.

08 · Bottom Line

PageIndex does one thing better than vector RAG:
it retrieves the right section, not the similar section.
The cost is ~133 LLM calls per 100 pages at build time,
with a failure mode that corrupts the entire tree — not just one result.

Use it for long, structured, high-stakes documents where retrieval accuracy matters more than indexing speed. Don't use it as a drop-in replacement for every RAG use case.
